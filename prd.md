# Mory Survey Site PRD

## 1. 문서 목적

이 문서는 현재 `mory-survey-site` 프로젝트를 기준으로, 다른 AI나 다른 개발자가 거의 동일한 결과물을 재현할 수 있도록 만드는 제품 요구 문서다.  
단순 아이디어 문서가 아니라, 현재 구현된 화면 구조, 동작 규칙, 디자인 시스템, 데이터 흐름, 관리자 기능, 환경변수 조건까지 포함한 재현용 스펙 문서로 사용한다.

문서 기준일: 2026-03-24  
프로젝트 성격: 한국 장례문화의 불편했던 순간을 수집하고 시각화하는 설문/아카이브 웹앱  
브랜드명: `Mory`

## 2. 제품 한 줄 정의

`Mory`는 "현재 장례문화가 불편할 때, 언제였나요?"라는 질문을 중심으로 사람들의 경험을 익명으로 수집하고, 승인된 의견을 공개 카드와 랭킹 형태로 보여주는 장례문화 인식 개선용 설문 사이트다.

## 3. 제품 배경과 문제 정의

현재 제품은 아래 문제의식에서 출발한다.

- 한국의 전통 장례문화는 획일적이고 형식적이며, 고인의 개별성과 조문객의 감정을 충분히 반영하지 못한다.
- 무거운 분위기와 엄숙함의 강요로 인해 특히 젊은 세대가 장례 공간에 거리감을 느낀다.
- 장례가 끝난 뒤 기억을 이어갈 디지털 구조가 부족하다.
- 종이, 조화, 일회용품, 불투명한 비용 구조 등 환경적/경제적 부담이 크다.
- 디지털 네이티브 세대에게 맞는 참여 방식이 부족하다.

이 설문 사이트는 장례문화 자체를 즉시 바꾸는 서비스가 아니라, 장례문화의 실제 불편 지점을 구조화해서 보여주고, 이후 `Memory Stone`, `Memory Wall`, `Mori App` 같은 확장 서비스의 인사이트 기반을 만드는 역할을 한다.

## 4. 제품 목표

### 핵심 목표

- 사용자가 장례문화에서 불편했던 순간을 짧고 부담 없이 남기게 한다.
- 수집된 의견을 승인 후 공개 카드로 보여주며 문제를 시각적으로 드러낸다.
- 카테고리 기반 랭킹을 통해 어떤 문제가 많이 반복되는지 보여준다.
- 관리자만 데이터베이스를 열람, 수정, 승인, 삭제, 수동 입력할 수 있게 한다.
- 전체 UX는 차분하고 품위 있으며, 과장되지 않은 추모/기억의 톤을 유지한다.

### 비핵심 목표

- 회원가입/로그인 시스템
- 복잡한 퍼블릭 프로필
- 댓글/좋아요/소셜 피드
- 고급 통계 대시보드
- 다국어 지원
- 실시간 협업

## 5. 타깃 사용자

### 1차 사용자

- 한국 장례문화에서 불편함을 느껴본 일반 사용자
- 특히 20~40대 디지털 네이티브 세대
- 장례 절차, 분위기, 비용, 디지털 부재, 접근성 문제에 의견이 있는 사람

### 2차 사용자

- 관리자 또는 운영자
- 수집된 의견을 검수하고 공개 상태를 관리하는 사람
- 리서치 기반 의견을 수동으로 등록하는 사람

## 6. 핵심 사용자 시나리오

### 공개 사용자

1. 메인 페이지에 들어온다.
2. 질문 "장례문화가 불편할 때, 언제였나요?"를 본다.
3. 텍스트 입력창에 경험을 작성한다.
4. 제출하면 데이터는 `pending` 상태로 저장된다.
5. 관리자가 승인하면 해당 의견이 메인 카드 영역과 랭킹에 반영된다.

### 관리자

1. `/admin`에 접속한다.
2. 비밀번호를 입력해 인증한다.
3. 전체 의견 목록을 열람한다.
4. 카테고리/태그를 보정하고 승인하거나, 삭제하거나, 대기중으로 돌린다.
5. 필요하면 의견을 수동으로 직접 등록한다.

## 7. 기술 스택

- 프레임워크: `Next.js 16` App Router
- 언어: `TypeScript`
- UI: `React 19`
- DB: `Supabase`
- 서버 권한 작업: `Supabase Service Role Key`
- 자동 분류: `Gemini API` 선택 사용
- 애니메이션: `framer-motion` 사용 가능
- 배포 대상: `Vercel`

## 8. 라우트 구조

### 페이지 라우트

| 경로 | 역할 | 렌더링 방식 |
| --- | --- | --- |
| `/` | 메인 설문 페이지 + 승인 의견 카드 목록 | 서버 컴포넌트 |
| `/rankings` | 승인 의견을 카테고리별로 묶어 보여주는 TOP 20 페이지 | 클라이언트 컴포넌트 |
| `/admin` | 관리자 인증 후 DB 열람/편집 페이지 | 서버 컴포넌트 + 클라이언트 컴포넌트 |

### API 라우트

| 경로 | 메서드 | 역할 |
| --- | --- | --- |
| `/api/opinions` | `GET` | 승인된 의견 목록 반환 |
| `/api/opinions` | `POST` | 공개 의견 등록, 기본 상태는 `pending` |
| `/api/opinions/[id]` | `PUT` | 관리자 권한으로 의견 수정/승인 |
| `/api/opinions/[id]` | `DELETE` | 관리자 권한으로 의견 삭제 |
| `/api/admin/auth` | `GET` | 관리자 인증 여부 확인 |
| `/api/admin/auth` | `POST` | 관리자 로그인 및 쿠키 발급 |
| `/api/admin/auth` | `DELETE` | 관리자 로그아웃 |
| `/api/admin/opinions` | `GET` | 관리자 권한으로 전체 의견 조회 |
| `/api/admin/opinions` | `POST` | 관리자 수동 입력 |
| `/api/admin/opinions` | `PATCH` | 관리자 상태 수정 |

## 9. 전역 UX 원칙

- 분위기는 차분하고 따뜻해야 한다.
- 과도하게 장난스럽거나 게임화된 표현은 피한다.
- 배경은 순백이 아니라 약간 크림 톤의 오프화이트를 사용한다.
- 무채색이 아니라 "따뜻한 돌/흙/기억" 계열의 브라운-스톤 팔레트를 사용한다.
- 정보 구조는 단순해야 한다.
- 슬픔을 과장하거나 우울함을 시각적으로 강요하지 않는다.
- "추모", "기억", "공감", "재정의"의 톤을 유지한다.

## 12. 페이지 상세 명세

### 12.1 메인 페이지 `/`

#### 목적

- 메인 질문 제시
- 의견 입력 유도
- 승인된 의견 카드 공개

#### 레이아웃

- 데스크톱: 좌측 고정 컬럼 + 우측 카드 컬럼
- 모바일: 세로 스택
- `1024px` 이상에서 2열 구조

#### 좌측 컬럼

- 메인 타이틀:
  - `장례문화가 불편할 때,`
  - `언제였나요?`
- 타이틀은 `Nanum Myeongjo` 사용
- 타이틀 아래에 인라인 입력 폼 배치

#### 입력 폼 요구사항

- 단일 textarea
- placeholder: `어떤 점이 불편하셨나요? 자유롭게 남겨주세요.`
- 최대 글자 수: `300`
- 글자 수 카운터 표시
- 버튼 라벨:
  - 기본: `의견 남기기`
  - 제출 중: `전송 중...`
- 성공 시 감사 메시지 표시:
  - 제목: `✓ 소중한 의견 감사합니다`
  - 설명: `남겨주신 이야기는 검토 후 모리 서비스에 반영하도록 할게요.`
- 실패 시 에러 문구 표시

#### 입력 처리 규칙

- 빈 문자열 제출 금지
- `POST /api/opinions` 호출
- 저장 성공 시 폼 리셋
- 공개 즉시 반영되지 않음
- 기본 상태는 `pending`

#### 우측 카드 영역

- 승인된 의견만 노출
- 메인 페이지는 서버에서 직접 Supabase 조회
- 조회 정렬 기준: `created_at ASC`
- 카드 번호 규칙:
  - 가장 먼저 등록된 의견이 `1번째 불편함`
  - 그 다음이 `2번째 불편함`
- 카드 상단에는 순환 이모지 사용
  - `🤔`, `💬`, `💡`, `👀`, `🙌`
- 카드 본문에는 의견 텍스트만 표시

#### 카드 레이아웃 규칙

- Masonry 유사 다단 컬럼
- 컬럼 수:
  - 기본 1열
  - `768px+`: 2열
  - `1280px+`: 3열
  - `1536px+`: 4열
- 카드 hover 시 약간 위로 이동

#### 스크롤 규칙

- 승인 의견이 5개 초과면 우측 컬럼에 스크롤 마스크 적용
- 높이: `calc(100vh - 8rem)`

#### 빈 상태

- 제목: `아직 승인된 의견이 없습니다.`
- 설명: `새로운 의견이 승인되면 이 영역에 카드가 표시됩니다.`

### 12.2 랭킹 페이지 `/rankings`

#### 목적

- 수집된 승인 의견을 카테고리별로 묶어 "어떤 문제가 많이 반복되는지" 보여준다.

#### 레이아웃

- 메인과 거의 동일한 2열 구조
- 좌측은 설명, 우측은 리스트

#### 좌측 콘텐츠

- 제목:
  - `장례식이 불편한 순간`
  - `TOP 20`
- 설명:
  - `그동안 사람들이 가장 많이 공감한 의견이에요.`
  - `여러분이 보여주신 관심 잊지 않고,`
  - `적극적으로 개선하겠습니다.`

#### 데이터 로딩

- 클라이언트에서 `/api/opinions` fetch
- API는 `approved` 상태만 반환

#### 그룹화 규칙

- `category` 기준으로 그룹화
- 그룹별 `count` 계산
- `count DESC` 기준 정렬
- 상위 20개 그룹만 노출

#### 대표 문장(summary) 생성 규칙

각 카테고리 그룹에서 대표 문장 1개를 뽑는다.

- 각 의견의 토큰을 분리한다.
- 다른 의견들과의 공통 토큰 수를 계산한다.
- 해시태그 공유에는 더 높은 가중치(`3배`)를 준다.
- 최종 점수가 높은 의견을 대표 문장으로 채택한다.
- 동점이면 더 긴 문장을 우선한다.

#### 리스트 아이템 구성

- 좌측: `1위`, `2위` 등 순위
- 중앙:
  - 대표 문장
  - `OO 관련 의견` 메타 텍스트
- 우측: `N명`

#### 빈 상태

- `데이터를 불러오는 중이거나 없습니다.`

### 12.3 관리자 로그인 페이지 `/admin`

#### 목적

- 관리자 비밀번호로 보호된 데이터 열람 입구

#### 구조

- 공개 사이트와 동일한 상단 헤더 유지
- 중앙 정렬된 로그인 카드 1개

#### 카드 구성

- eyebrow: `Protected Admin Page`
- 제목: `데이터베이스 열람 페이지`
- 설명: `저장된 의견 데이터를 확인하거나 수동으로 등록하려면 관리자 비밀번호가 필요합니다.`
- 입력 라벨: `PASSWORD`
- CTA:
  - 기본: `페이지 열기`
  - 제출 중: `확인 중...`

### 12.4 관리자 대시보드 `/admin`

#### 목적

- 전체 데이터 열람
- 상태 승인/대기 전환
- 삭제
- 카테고리 및 해시태그 수정
- 관리자 수동 입력

#### 상단

- 제목: `관리자 데이터베이스`
- 설명: `입력 폼에서 쌓인 데이터를 열람하고, 상태를 변경하거나 관리자 데이터를 직접 추가할 수 있습니다.`
- 우측 상단 로그아웃 버튼

#### 관리자 수동 입력 영역

- textarea: 의견 본문
- input 1: 카테고리
- input 2: 해시태그(쉼표 구분)
- 버튼: `직접 등록`
- 기본 등록 상태: `approved`
- 카테고리 미입력 시 `관리자 리서치`

#### 데이터 테이블 컬럼

- 상태
- 내용
- 카테고리
- 태그(쉼표 구분)
- 등록일
- 관리

#### 상태 표시

- `pending`: 노랑 배지, 텍스트 `대기중`
- `approved`: 초록 배지, 텍스트 `승인됨`
- `rejected`: 빨강 배지, 텍스트 `거절됨`

#### 관리 액션 규칙

`pending` 행:

- `저장 및 승인`
- `삭제`

`approved` 또는 `rejected` 행:

- `수정 저장`
- `대기중으로 변경`
- `삭제`

#### 날짜 표시

- `ko-KR` 로케일의 `toLocaleString`

## 13. 데이터 모델

### 테이블: `public.opinions`

| 필드 | 타입 | 규칙 |
| --- | --- | --- |
| `id` | `uuid` | PK, 기본값 `gen_random_uuid()` |
| `content` | `text` | 필수 |
| `category` | `text` | 필수, 기본값 `기타` |
| `hashtags` | `text[]` | 필수, 기본값 빈 배열 |
| `status` | `text` | `pending`, `approved`, `rejected` 중 하나 |
| `created_at` | `timestamptz` | UTC now 기본값 |

### 상태 정의

- `pending`: 공개 미노출, 관리자 검수 대기
- `approved`: 메인/랭킹에 공개 가능
- `rejected`: 저장은 유지하되 공개 제외 가능 상태

## 14. 카테고리 체계

허용 카테고리 목록:

- `분위기`
- `편의성`
- `추모방식`
- `절차`
- `환경/비용`
- `의식`
- `식문화`
- `의복`
- `트렌드`
- `소통`
- `기타`
- `비용`
- `인프라`
- `행정`
- `답례풍습`
- `사회인식`

규칙:

- 유효하지 않은 카테고리는 모두 `기타`로 정규화한다.
- 해시태그는 배열로 저장한다.
- 해시태그는 `#`를 제거하고 공백 없이 정규화한다.
- 최대 3개까지만 유지한다.
- 각 해시태그는 최대 24자까지 허용한다.

## 15. 인증과 보안

### 관리자 인증 방식

- 별도의 사용자 테이블 없음
- 단일 관리자 비밀번호 방식
- 로그인 성공 시 쿠키 발급

### 인증 세부 규칙

- 쿠키명: `mory_admin_session`
- 세션 토큰은 `ADMIN_PASSWORD`와 `ADMIN_SESSION_SECRET` 기반 SHA-256 해시
- 쿠키 속성:
  - `httpOnly`
  - `sameSite: lax`
  - `secure: production only`
  - `path: /`
  - `maxAge: 12시간`

### 기본값

- 기본 관리자 비밀번호: `mory1234!`
- 기본 세션 시크릿: `mory-admin-session-v1`

운영에서는 반드시 환경변수로 덮어써야 한다.

## 16. API 동작 상세

### `GET /api/opinions`

- 공개용 승인 의견 목록 반환
- 정렬: `created_at DESC`
- DB 미연결 시 mock 데이터 반환

### `POST /api/opinions`

- 입력: `content`, 선택적으로 `category`, `hashtags`
- 빈 content 금지
- 저장 시:
  - `status = pending`
  - `category/hashtags`는 우선 정규화 저장
- DB 연결이 없으면 `503`

### Gemini 자동 분류

Gemini API가 연결되어 있고 관리자용 Supabase 클라이언트가 있으면:

1. 의견을 우선 DB에 저장
2. 백그라운드(after)에서 Gemini 분류 실행
3. category/hashtags만 후속 업데이트

Gemini 분류 규칙:

- 카테고리는 허용 목록 중 정확히 1개
- 해시태그는 2~3개
- 짧은 한국어 표현
- `#` 없이 저장

### `PUT /api/opinions/[id]`

- 관리자 쿠키 필요
- 수정 가능 필드:
  - `status`
  - `category`
  - `hashtags`
  - `content`

### `DELETE /api/opinions/[id]`

- 관리자 쿠키 필요
- 해당 row 삭제

### `GET /api/admin/opinions`

- 관리자 쿠키 필요
- 전체 의견 반환
- 정렬: `created_at DESC`

### `POST /api/admin/opinions`

- 관리자 쿠키 필요
- 관리자가 직접 의견 생성
- 기본 카테고리: `관리자 리서치`
- 기본 상태: `approved`

## 17. Supabase 정책 요구사항

현재 기준 필요한 RLS 정책:

- 공개 사용자(`anon`, `authenticated`)는 `insert` 가능
- 공개 사용자는 `status = approved`인 row만 `select` 가능
- 관리자 작업은 `service_role` 클라이언트로 우회 수행

즉, 관리자 페이지의 전체 열람/수정/삭제는 반드시 `SUPABASE_SERVICE_ROLE_KEY` 기반 서버 클라이언트를 사용해야 한다.

## 20. 접근성/사용성 요구사항

- 모든 주요 로고/링크는 키보드 접근 가능해야 한다.
- 관리자 페이지는 비밀번호 입력 후 엔터 제출 가능해야 한다.
- 텍스트 입력은 명확한 placeholder와 상태 메시지를 가져야 한다.
- 에러 메시지는 붉은 텍스트로 표시해야 한다.
- 버튼 disabled 상태는 시각적으로 구분 가능해야 한다.
- 긴 의견 텍스트는 줄바꿈과 word-break가 가능해야 한다.

## 21. 콘텐츠 톤 가이드

- 차분하고 공감적인 문체
- 지나치게 공식적이거나 관료적인 표현 지양
- 과장된 감정 유도 금지
- 슬픔을 강요하지 않고 "기억, 위로, 개선"의 방향을 유지
- 관리자 화면도 과하게 차갑지 않게 동일한 브랜드 톤 유지

## 22. 구현 시 유지해야 하는 현재 프로젝트의 특징

다른 AI가 재현할 때 아래 특징은 반드시 유지하는 것을 권장한다.

1. 메인 페이지는 카드 목록을 "인기순"이 아니라 `등록일 오름차순` 기준 번호로 표시한다.
2. 랭킹 페이지는 의견 자체를 그대로 순위화하지 않고, `카테고리 그룹`을 순위화한다.
3. 공개 제출은 곧바로 노출되지 않고 반드시 `pending`을 거친다.
4. 관리자 페이지는 단일 비밀번호 보호 방식이다.
5. 공개 사이트와 관리자 사이트는 같은 브랜드 헤더를 공유한다.
6. 메인/랭킹의 대제목에는 `Nanum Myeongjo`, 나머지 텍스트에는 `Pretendard`를 쓴다.
7. 전체 UI는 흰색과 회색이 아니라 "따뜻한 브라운-스톤" 팔레트를 유지한다.

## 24. 알려진 구현 메모

- `components/OpinionForm.tsx`와 `OpinionForm.module.css`는 현재 메인 UX의 주력 입력 폼이 아니다.
  - 실제 메인 입력은 `InlineOpinionForm.tsx`가 담당한다.
- 로고 자산으로 `logo.png`도 존재하지만, 현재 활성 구현은 `logo_mori_fill.svg`를 사용한다.
- DB가 완전히 미연결된 경우 공개 `GET`은 mock 데이터를 반환하지만, 공개 `POST`는 저장 불가로 `503`을 반환한다.
- 관리자 전체 기능은 `SUPABASE_SERVICE_ROLE_KEY` 없이는 정상 동작하지 않는다.

## 25. 재현 수용 기준

아래 조건을 만족하면 현재 프로젝트와 실질적으로 동일한 제품으로 본다.

### 기능 기준

- 메인에서 의견 제출 가능
- 제출 데이터는 DB에 `pending`으로 저장
- 승인된 의견만 메인 카드와 랭킹에서 노출
- 메인 카드 번호는 등록일 오름차순 기준
- `/admin`에서 비밀번호로 접근 가능
- `/admin`에서 전체 목록 조회, 승인, 삭제, 수정, 수동 입력 가능

### 디자인 기준

- 크림 배경 + 웜 브라운 텍스트 팔레트 유지
- 헤더에 로고와 브랜드 라벨 존재
- 메인/랭킹 타이틀은 세리프 계열(`Nanum Myeongjo`)로 강조
- 카드와 입력 요소는 둥근 모서리와 얕은 그림자 유지
- 전체 인상은 차분하고 프리미엄한 추모 서비스 톤이어야 함

### 데이터 기준

- `opinions` 테이블 구조 동일
- status enum 성격 동일
- 카테고리 정규화 규칙 동일
- 해시태그 최대 3개 규칙 동일

## 26. 향후 확장 방향

현재 설문 사이트는 Mory 전체 서비스 생태계의 인사이트 수집 레이어다. 이후 확장 시 아래 방향과 자연스럽게 연결되어야 한다.

- `Memory Stone`: 입구용 디지털 비석
- `Memory Wall`: 현장 대형 추모 디스플레이
- `Mori App`: 장례 이후에도 기억을 이어가는 디지털 추모관

즉, 현재 설문 사이트도 완전히 별개의 웹앱처럼 보이기보다, 장례 경험을 재정의하는 더 큰 브랜드의 출발점처럼 느껴져야 한다.
